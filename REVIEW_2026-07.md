### A2. Architectural debt

- **MED — `agent/api.py` is still a ~3000-line god-module.** `Agent.__init__`
  runs lines 173–716 doing 15+ subsystem resolutions inline. Extract cohesive
  `_wire_workspace` / `_wire_living_plan` / `_wire_stop_hooks` helpers returning
  small structs so the constructor reads as composition.
- **MED — `Dependencies` is a 31-field struct; the `fast_*` flags leak internals.**
  Eight booleans encode "is this layer a no-op default?" as denormalized state
  that `_loop` must keep in sync (`api.py:1917-1943`). The invariant
  `fast_permissions == isinstance(permissions, AllowAll)` lives in `_loop`, not
  on the struct — a future non-`AllowAll` default could silently desync and skip
  real permission checks. Derive fast-ness from a marker attribute on default
  impls, or group into a nested `FastFlags`.

### A4. Efficiency

- `vectorstore/inmemory.py:200-205` — O(N·D) full scan re-normalizing every
  stored vector every query; cache per-vector norms at `add` time.
- `memory/redis_facts.py:273-287` — fact store N+1 (`RedisMemory` was pipelined,

### A5. Effectiveness of the 12 architectures (current)

**Production-grade in structure:** ReAct (parallel dispatch correct, slot-indexed
results, timeout + summarize + hard-cap tiers), ReWOO (real topological parallel
+ cycle detection), ToT (real BFS-beam + min-score prune + synthesis), Supervisor

## Part B — Competitive gap analysis (2026)

Sourced from a fanned-out web-research pass (24 sources fetched, 25 claims
adversarially verified, 24 confirmed / 1 refuted) across LangGraph, CrewAI,
AutoGen/AG2, OpenAI Agents SDK, Google ADK, Pydantic AI, LlamaIndex Workflows,
Mastra, Claude Agent SDK, Temporal/Restate/DBOS stacks, DSPy, and the
Mem0/Zep/Letta/Cognee memory layer. Key primary sources: Anthropic engineering
(advanced tool use, code execution with MCP), Cloudflare (Code Mode),
OpenTelemetry GenAI/agent semantic conventions, LangGraph docs, arXiv 2602.19320
(agent memory taxonomy). *Caveats: figures are vendor best-case; OTel GenAI/MCP
conventions are still "Development" status; one comparative claim about OpenAI's
HITL was refuted (OpenAI Agents SDK **does** ship first-class approval
primitives).*

### B1. The 7 axes — table-stakes vs frontier, and where Loom stands

Legend: ✅ have it · 🟡 partial · ❌ missing · 🏆 ahead of the pack

#### (1) Orchestration primitives
- **Table-stakes (2026):** an agent loop with tool calls + **handoffs** until a
  stop condition (OpenAI SDK model); a **graph** of nodes/edges/checkpoints
  (LangGraph); **role-based** declarative crews (CrewAI). MCP + increasingly A2A
  as connective protocols.
- **Frontier:** dynamic agent spawning; A2A (agent-to-agent) protocol support.
- **Loom:** 🏆 **12 architectures behind one `Agent` kwarg** is genuinely more
  than any single competitor ships. `Workflow` (DAG) + `Team` cover graph and
  role models. **Gaps:** ❌ no dynamic runtime sub-agent spawning (workers are
  pre-declared — same limitation flagged in Part A); ❌ no A2A protocol;
  handoffs exist (Swarm) but carry flat-string state, not structured artifacts.

#### (2) Context / memory management
- **Table-stakes:** the CoALA 4-type taxonomy — **working / episodic / semantic /
  procedural** memory; automatic **compaction** at a token watermark (Claude Code
  compacts at `auto_compact_limit`; best practice = sync at ~64% context, hand off
  at ~80%); **multi-signal retrieval fusion** (semantic + keyword + entity scored
  in parallel and fused).
- **Frontier:** **temporal knowledge graphs** with fact-validity windows (Zep's
  Graphiti scores 63.8% on LongMemEval vs Mem0's 49.0% — a ~15pt edge on temporal
  retrieval); **observation masking beats LLM summarization** for compaction
  (JetBrains: +2.6% solve rate, 52% cheaper); entity-linking that influences
  ranking (the industry moved *away* from directly-traversable graph stores);
  **procedural memory** (learned skills/rules).
- **Loom:** 🏆 has **bi-temporal facts** (valid-time vs ingestion-time) — this is
  the Zep-class differentiator most frameworks lack, and it's already shipped.
  ✅ episodic + semantic + working blocks, hybrid BM25+vector+RRF recall (matches
  the "multi-signal fusion" best practice). **Gaps:** 🟡 compaction fires only
  *between* architecture passes, not at a mid-loop token watermark (Part A);
  ❌ **no token-budgeted memory injection** (limit is a row count, not tokens);
  ❌ **no recency/temporal decay** on recall ranking; ❌ **no procedural memory**;
  🟡 facts are SPO triples with no multi-hop traversal.

#### (3) Durability & human-in-the-loop
- **Table-stakes:** **checkpoint / resume**; **HITL interrupts** (pause → return
  pending approvals → resume with state). LangGraph is the reference (checkpoint
  every node, `interrupt()`, **time-travel** = rewind/edit/fork). OpenAI SDK ships
  `needsApproval` + `RunState` resume. **True durability > chat history:** an
  execution **journal**, **idempotent tool boundaries**, **versioned prompts/
  tools**, **durable approvals**, and crash-recovery that never repeats side
  effects (Temporal/Restate/DBOS).
- **Frontier:** time-travel debugging (rewind to any checkpoint, edit state,
  fork); deterministic replay with versioned model+prompt.
- **Loom:** ✅ has the **journal + idempotency-key + input-fingerprint** pieces
  (just hardened) — that's the "beyond chat-history" bar most frameworks miss.
  **Gaps:** 🟡 HITL is **allow/deny only** — no "edit args and continue," no
  "inject guidance," no pause/resume mid-run (Part A); ❌ **no time-travel /
  fork**; 🟡 resume replays step results but doesn't snapshot `session.messages`,
  so a mid-architecture crash restarts the architecture; ❌ no prompt/tool
  **versioning** for deterministic replay.

#### (4) Model-layer features
- **Table-stakes:** **prompt caching**; **extended thinking / reasoning**;
  **structured outputs**; streaming.
- **Frontier:** **streaming tool-call argument deltas**; **model fallback chains**
  (failover to a secondary model/provider on permanent error or sustained rate
  limit); **per-request timeouts**; OpenAI **Responses API**; **Batch API**.
- **Loom:** ✅ prompt caching (message-tail breakpoint, just added), ✅ extended
  thinking with correct signature round-trip (just added), ✅ robust structured
  output (unions, fence-strip, validation-retry), ✅ token/milestone streaming.
  **Gaps:** ❌ **no model fallback chain** (RetryingModel retries the *same*
  model); ❌ **no per-request wall-clock timeout** (a hung stream blocks
  forever); ❌ no tool-call-arg-delta streaming; ❌ no Responses API path; ❌ no
  Batch API seam; ❌ no embeddings on the Model protocol.

#### (5) Tool ecosystem & MCP  — *biggest strategic gap*
- **Table-stakes:** native **MCP**. But MCP definitions **bloat context**: 7
  servers = 67.3K tokens (33.7% of a 200K window) before any input; GitHub's MCP
  alone = ~18K tokens; enterprise = 134K before optimization.
- **Frontier (this is where the field moved in late 2025 / early 2026):**
  - **Tool Search / deferred loading** — Anthropic's `defer_loading` flag +
    tool-search: **85% token reduction** with full library access, and accuracy
    *gains* (Opus 4.5: 79.5% → 88.1%). Claude Code auto-activates it when tool
    descriptions exceed 10% of the context budget.
  - **Code Mode / code-execution-with-MCP** — agents write code that calls MCP
    methods in a **sandbox**, returning only results: Anthropic 150K → 2K tokens
    (**98.7%**); Cloudflare's V8-sandbox Code Mode: 1.17M-token API surface → ~1K
    (**99.9%**).
  - **Computer use.**
- **Loom:** ✅ MCP-native (client hardened this pass), ✅ skills give progressive
  disclosure *for instructions*. **Gaps — the strategic ones:**
  ❌ **no tool search / deferred tool loading** (ships every tool schema every
  turn — the exact 67K-token bloat problem the frontier just solved);
  ❌ **no code-execution sandbox seam** for tools (bash runs on host behind a
  bypassable denylist; no Docker/E2B/V8 executor interface);
  ❌ no computer-use surface; ❌ no A2A. *This axis is where Loom is furthest
  behind the 2026 frontier and where the highest-leverage roadmap items live.*

#### (6) Observability & evals
- **Table-stakes:** **OpenTelemetry GenAI semantic conventions** — spans with
  `gen_ai.operation.name`, `gen_ai.provider.name`, `gen_ai.request.model`,
  `gen_ai.usage.input_tokens/output_tokens`; MCP spans with `mcp.method.name` /
  `mcp.session.id`; W3C trace-context linking client↔server. Native support in
  Braintrust/Langfuse/Phoenix for 20+ frameworks *via the OTel fallback*.
- **Frontier:** typed agent-trace schema (tool calls / reasoning / state
  transitions / memory ops as parent-child spans across handoffs); **online eval**
  (scorers attached to live production traces); agent-eval harnesses.
- **Loom:** ✅ OTel spans + metrics + zero-cost disabled path. **Gaps:**
  ❌ **spans do NOT follow `gen_ai.*` conventions** (free-form `loom.*` names) —
  so Loom traces won't light up in Braintrust/Langfuse/Datadog/Phoenix GenAI
  dashboards out of the box, the single highest-ROI observability fix;
  ❌ no MCP `mcp.*` span attributes; ❌ **no agent-eval / benchmark harness** in
  the repo (no golden-task suite, no online-eval seam).

#### (7) Production concerns
- **Table-stakes:** **multi-tenancy**, **cost governance**, retries/idempotency,
  streaming/sessions; managed deploy platforms (LangGraph Platform, CrewAI+)
  supply these.
- **Frontier:** **per-tenant QPS rate limiting**; deployment/runtime.
- **Loom:** 🏆 **`user_id` as a first-class typed primitive** partitioning memory
  + budgets is genuinely ahead — most frameworks make tenancy a config string you
  can typo. ✅ cumulative token/cost/wall-clock budgets, correct enforcement
  ordering. **Gaps:** ❌ **no per-tenant QPS / token-bucket rate limiting** (only
  cumulative caps — a tenant can burst until their *cumulative* cap trips); ❌ no
  deployment/runtime story (library only); ⚠️ the Part-A cross-tenant `handle.touch`
  bug *undermines the tenancy pitch until fixed*.

## Combined recommendation (Parts A + B)

**Two independent tracks, both worth doing:**

**Track 1 — Fix the A1 correctness bugs (small, safety-critical, do first).**
The cross-tenant `handle.touch` leak (#1), fail-open destructive gate (#2),
ungated Reflexion/Swarm budgets (#3/#4), Chroma/FAISS score-contract breaks
(#8/#9), and the Redis anon/time-range recall bugs (#10) are wrong-answer/safety
defects. They also directly undermine two of Loom's *leadership* claims
(multi-tenancy, memory quality). Independently testable; ~1–2 days.

**Track 2 — Close the frontier gaps that make Loom "world's best" (ranked).**
1. **Tool Search / deferred tool loading** (B2 #1) — the field's biggest 2026
   shift; Loom already has `list_tools(query=)` plumbing and a skills
   progressive-disclosure pattern to build on.
2. **Code-execution sandbox seam** (B2 #2) — an `executor=` interface
   (Docker/E2B/V8) that also fixes bash security.
3. **OTel `gen_ai.*` conventions** (B2 #3) — cheap, high-ROI ecosystem unlock.
4. **Pause/interject/edit-args HITL + message-level checkpoint** (B2 #4).
5. **Token-budgeted + decayed memory injection + mid-loop compaction** (B2 #5).
6. **Per-tenant QPS limiter, model fallback chains, eval harness** (B2 #6–8).

Loom's foundations (breadth, bi-temporal memory, typed tenancy, durability) are
genuinely leadership-grade. The gap to "world's best" is **tool-context
efficiency (Tool Search + Code Mode), observability-standard compliance, and
HITL depth** — plus fixing the handful of correctness bugs that currently
undercut the strengths it already has.
